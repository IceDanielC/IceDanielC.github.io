# 项目介绍

# 背景

目前 planning 产线已经有不少通过迭代 mining 逻辑（好数据 / 坏数据）获得实车效果提升的案例，但数据产线还缺乏对于 mining 体系的抽象整合及系统化支持，Planning Mining Framework（PMF）旨在为 planning 数据挖掘工作提供系统化的抽象，使得数据挖掘过程尽可能体系化 / 自动化整合，最终提升数据迭代的飞轮效应。
Planning Mining Framework（PMF）作为 Planning 训练数据挖掘框架，为数据产线好数据挖掘、坏数据清洗提供统一的底层编程框架支持；其构建于 PPL 底层编排，数据湖存储基础之上，为 planning 数据挖掘工作提供系统化的抽象。

# 概念

Dataset: 抽象得表示一组特定目的数据 event 的集合，对应 PMF 中的 Dataset class。
Transform: 抽象得表示一组对于数据的操作逻辑，对应 PMF 中由若干 Operator class 组成的计算 DAG 逻辑。
**Mining 过程：** 通过在特定 Dataset 上运用 Transform（特定 mining 逻辑）获取新的 Dataset。

# type 和 interface 的区别

  1. 接口使用 extends 继承，type 使用 & 实现继承
  2. interface 可以申明合并，type 不可以
  3. interface 可以使用 implements 实现一个接口在 class 中
  4. 几乎所有 interface 具有的功能，type 都可以实现，主要区别在于 type 不能重新打开类型来添加新成员，而接口总是可以扩展的，即便这样官方还是建议我们尽量去使用接口代替类型别名。另一方面，如果你无法通过接口来描述一个类型并且需要使用联合类型或元组类型，这时通常会使用类型别名。

# Pro-Components

ProTable 的诞生是为了解决项目中需要写很多 table 的样板代码的问题，所以在其中做了封装了很多常用的逻辑。这些封装可以简单的分类为预设行为与预设逻辑。

依托于 ProForm 的能力，ProForm 拥有多种形态，可以切换查询表单类型，设置变形成为一个简单的 Form 表单，执行新建等功能。

use-antd-resizable-header Github
在搜索列宽拖拽的解决方案时，在 ant-design 的 github 上看到这个

table 组件能否支持拖拽调整列宽（伸缩列）？・Issue #32594・ant-design/ant-design (github.com)

注意
不能给所有的 column 设置 width, 最后一列不能设置，否则将不能拖拽

总结
相比较下，react-resizable 在多列时会出现卡顿，use-antd-resizable-header 因为没有实时调整，而是松开鼠标后调整，就不会显得非常卡顿，而且写的代码更少，所以我选择 use-antd-resizable, 但其实都还有比较多的扩展可以实现，不过我是搞后端的，所以就这样吧.

# 第二周

移除不必要的 useMemo 和 useCallback
需要记住的最重要的一件事是， useMemo 和 useCallback 只有在重渲染的过程中才有用。在初始渲染过程中，它们不仅是无用的，甚至是有害的：它们会让 React 做很多额外的工作。这意味着你的应用在初始渲染过程中会稍稍更慢一些。并且，如果你的应用有数百个这些 hooks 分布在各处，那么这些轻微的影响初始渲染的作用就可以被观察到。
